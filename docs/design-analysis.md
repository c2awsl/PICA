# PICA 图片识别分组系统 — 完整设计分析

> 本文档分析当前系统的不足，提出完整的功能设计和交互方案。

---

## 一、现状诊断

### 当前数据模型

```
Image
├── suggested_category  (JSON list)        ← AI 建议分类
├── suggested_tags      (JSON list)        ← AI 建议标签
├── confirmed_category  (JSON list)        ← 用户确认分类
├── confirmed_tags      (JSON list)        ← 用户确认标签
├── work_name           (single string)    ← 作品/系列名称
├── image_type          (single string)    ← 图片类型
├── ai_status           (pending/processing/done/failed)
├── status              (pending/confirmed/rejected/skipped)
├── similar_group_id    → SimilarGroup
└── phash               (64-bit hex)
```

### 核心缺失

| 缺失能力 | 影响 |
|---------|------|
| 无层级分类（树状） | 无法精细管理：`动漫 > 作品 > 角色 > 场景` |
| 无用户标签体系 | 标签只能 AI 生成，不能统一管理、合并、删除 |
| 确认后不可编辑 | 归档后分类/标签永久锁定 |
| 无手动分组 | 只能按 phash 自动分组，无法人工干预 |
| 无多作品关联 | `work_name` 是单值，但一张图可能属于多个作品 |
| 搜索能力弱 | 仅支持 filename 模糊搜索和精确过滤 |
| 无批量操作（归档页） | 归档后无法批量修改 |
| 无 AI 置信度 | 用户看不到 AI 的可信度 |
| 统计页简陋 | 仅有计数，无趋势、分布、AI 效率等 |
| 队列管理空白 | 无法查看 AI 处理队列 |

## 二、数据模型改进

### 2.1 分类 → 三层级标签树

```
Category Tree (可配置、可拖拽排序)
├── 动漫/游戏
│   ├── 作品：原神
│   │   ├── 角色：雷电将军
│   │   └── 场景：璃月港
│   ├── 作品：明日方舟
│   └── 作品：孤独摇滚
├── 梗图/表情包
│   ├── 网络梗图
│   └── 自制表情包
├── 截图
│   ├── 聊天截图
│   ├── 游戏截图
│   └── 长截图
├── 艺术/设计
│   ├── 插画
│   ├── 官方美术
│   └── AI 生成
└── 生活/随拍
    ├── 人物
    ├── 风景
    ├── 动物
    └── 食物
```

**表结构：**
```sql
image_categories:
  id, image_id FK, category_id FK, level (ai_suggested=0/user_confirmed=1)

categories:
  id, parent_id FK, name, slug, sort_order, icon
```

这样图片可以同时属于多个分类节点，AI 建议和用户确认分离。

### 2.2 标签 → 统一标签库

```
tags: id, name, color, usage_count
image_tags: image_id, tag_id, level (suggested/confirmed)
```

- 标签全局统一：合并同义词、删除脏标签
- 标签带颜色：视觉分组
- 使用计数：展示热门标签

### 2.3 作品 → 多值 + 元数据

```sql
works: id, name, type (anime/game/manga/novel), aliases JSON
image_works: image_id, work_id, level (suggested/confirmed)
```

- 一张图片关联多个作品（如：原神×KFC 联动的同人图）
- 作品别名（原神 / Genshin Impact / げんしん）
- AI 自动匹配已有作品，给出建议

### 2.4 分组 → 人工+自动混合

```sql
similar_groups:
  id, phash_ref, name, is_auto (boolean), created_at
  images → Image.similar_group_id

manual_groups:
  id, name, description, created_at
  
manual_group_items:
  group_id, image_id, sort_order
```

- `is_auto`: 标记是自动生成还是手动创建
- 手动分组可跨状态（pending + confirmed 混合）
- 组内可排序

### 2.5 审计日志

```sql
image_logs:
  id, image_id, action, old_value JSON, new_value JSON,
  created_at, source (ai/user/batch)
```

所有变更可追溯：AI 改了啥、用户改了啥、批量改了啥。

## 三、AI 流程增强

### 3.1 多模型流水线

不再是单一 prompt 一把梭，而是流水线：

```
图片入库
  ├→ Step 1: 标签/分类识别（当前 prompt，提取 category+tags+work+type）
  ├→ Step 2: 质量过滤
  │   ├→ 模糊检测 (Laplacian variance)
  │   ├→ 暗图/过曝检测
  │   └→ 低分辨率标记
  ├→ Step 3: NSFW 检测（可选，敏感内容标记）
  └→ Step 4: OCR（可选，提取图片文字用于搜索）
```

每个步骤可独立启停、独立配置模型。

### 3.2 AI 置信度

```json
{
  "category": [
    {"name": "梗图", "confidence": 0.92},
    {"name": "游戏截图", "confidence": 0.45}
  ],
  "work": {"name": "原神", "confidence": 0.88},
  "type": {"name": "screenshot", "confidence": 0.76},
  "tags": [
    {"name": "派蒙", "confidence": 0.91},
    {"name": "对话", "confidence": 0.83}
  ]
}
```

- UI 展示置信度百分比或星级
- 仅高置信度（>0.8）自动填充，其余需人工确认
- 阈值可在设置页调节

### 3.3 AI 队列管理

- 展示 AI 队列长度、等待时间、处理速度
- 支持重新排序、取消单个任务、优先处理某个图片
- 实时显示当前正在处理的图片

## 五、操作逻辑与交互流程

### 5.1 核心工作流

```
同步 → 扫描 → AI识别 → 待办 → 确认 → 归档
                          ↓
                       驳回/跳过
                          ↓
                       可恢复回收站（30天）
```

### 5.2 确认归档流程（重新设计）

当前：点击 ✓ → 直接归档（用 AI 建议的分类）
改进：

```
点击 [归档]
  ├→ 弹出浮窗：选择分类（树形浏览 + 搜索）
  │   ├→ 保留 AI 建议（高亮显示，可直接点击选择）
  │   ├→ 自定义分类（树形导航或搜索）
  │   ├→ 编辑标签（从已有标签选择 + 新增）
  │   └→ 关联作品（从已有作品选择 + 新增）
  ├→ 点击归档 → 文件移动到 archive/{category}/
  └→ 快捷键: Enter = 确认归档, Esc = 取消
```

### 5.3 批量操作流程

```
选中多张图片
  ├→ 批量归档：统一指定分类（保留原标签差异）
  │   ├→ 所有图片同一分类
  │   ├→ 或批量移动（已归档的重新分类）
  │   └→ 追加/覆盖标签选项
  ├→ 批量驳回：确认 → 全部驳回
  ├→ 批量重试 AI：全部重置为 pending
  ├→ 批量编辑：追加标签、移除标签、修改类型
  └→ 批量导出/ZIP 打包
```
